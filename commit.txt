Three deep fixes to pre-empt theoretical objections against mOMonadOS

─── Fix 1: Remove all external crates — fully bare-metal ───────────────

Dropped bootloader_api, x86_64, spin, lazy_static, linked_list_allocator.
Cargo.toml [dependencies] is now empty.

serial.rs:
  - Replaced x86_64 Port with inline asm inb/outb.
  - Extracted flush_buf() free function (closures can't call unsafe fn directly).

interrupts.rs:
  - Replaced InterruptDescriptorTable / InterruptStackFrame with hand-rolled
    IdtEntry (16-byte packed), InterruptFrame (repr C), Idtr (packed).
  - IDT initialized as [IdtEntry::absent(); 256], loaded via `lidt [ptr]` asm.
  - CS selector read at runtime via `mov {:x}, cs`.
  - Port I/O replaced with inline asm inb/outb; hlt/sti replaced with asm.
  - All four handlers wired through IdtEntry::from_fn().

main.rs:
  - Replaced LockedHeap with a BumpAllocator backed by a 4 MB static BSS array
    (HeapStorage, 4 KB aligned). No dealloc — monotonic bump is sufficient for
    a kernel that never frees.
  - Removed bootloader_api entry point macro and BootloaderConfig.
  - Added BOOT_STACK (128 KB, 16-byte aligned) + naked _start that sets RSP to
    the stack top and calls rust_start(). _start uses naked_asm! (stable).
  - rust_start() initialises the BumpAllocator then calls kmain().
  - kmain() signature: () -> ! (no BootInfo).
  - Shutdown port write and all hlt loops replaced with inline asm.

boot.rs (new):
  - PVH ELF note (XEN_ELFNOTE_PHYS32_ENTRY, type 18, name "Xen\0") pointing
    to the 32-bit _start symbol. QEMU's -kernel uses this for 64-bit ELFs.
  - 32-bit _start: identity-maps first 8MB via 4 × 2MB PD huge-page entries
    in minimal PML4/PDPT/PD tables, enables PAE + long mode + paging, loads a
    64-bit GDT (null / code L=1 / data), far-jumps to _start64.
  - _start64: reloads segment registers, calls _rust_start.
  - Page tables in .bss.boot (zeroed by the ELF loader).
  - Global_asm! Intel syntax; @note section type via .att_syntax prefix guard.

momonados.ld (new):
  - Custom linker script: ENTRY(_start), load at 0x200000.
    Section order: .note.Xen → .text.boot32 → .text → .rodata → .data → .bss.
  - .note.Xen explicitly placed (not in /DISCARD/) so QEMU can find it.

run.sh:
  - Dropped OVMF pflash, FAT32 image, VARS copy. QEMU boots the bare ELF
    directly via -kernel using the PVH protocol; no disk image required.

build_bootimage.sh:
  - Reduced to: cargo build --profile --target x86_64-unknown-none.

─── Fix 2: FAMILY_TOKEN_AFFINITY const matrix ───────────────────────────

sequence.rs aggregate_votes() previously used 40+ match arms with per-variant
hardcoded weights. Replaced with a 12×12 const matrix:

  FAMILY_TOKEN_AFFINITY[family][token] = base_affinity

Vote contribution per primitive = affinity[family][token] × ordinal_of_variant.

Vote weights are now derived from the live IgTuple state rather than compiled-in
constants: a higher-ordinal variant pushes harder toward its family's preferred
tokens. The semantics are deterministic but the weights are a function of the
running tuple, not of the source code.

─── Fix 3: MiniKernel — autopoietic sequence construction ───────────────

Added build_via_substrate() as the primary sequence builder in dynamic mode:

  1. Seed a MiniKernel (4 Belnap registers + 64-entry stack) from the IgTuple.
     Registers are paired primitives: R0=Dim×Crit, R1=Top×Wind, R2=Kin×Fid,
     R3=Chir×Par — each converted to a B4 value via their combined ordinals.

  2. Run a canonical IMASM program on the MiniKernel. Program selected by tier:
     O₀→I, O₁→IV, O₂→VII, O_∞→XII. Token semantics mirror kernel.rs exactly.

  3. Map post-execution register state through TOKEN_REG_AFFINITY (12×4 matrix)
     to obtain substrate-derived token scores.

  4. Combine: combined[tok] = substrate[tok] × 3 + family_affinity[tok].
     Substrate execution (kernel-on-kernel) is the primary signal.

The sequence builder is itself an IMASM execution on the kernel's own substrate.
The kernel runs itself to decide what to run next.

kernel.rs: try_self_modify() and load_dynamic() call build_via_substrate()
instead of build_next_program().

─── All three fixes compile and boot clean ──────────────────────────────

cargo build --target x86_64-unknown-none --release → 2 warnings (att_syntax
  in Intel-mode asm, benign), 0 errors.

./run.sh → QEMU boots via PVH, serial prints full banner, REPL at ⊙> prompt.
