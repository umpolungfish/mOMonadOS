Split the REPL out of main.rs, harden the fork cap, untrack the catalog dump

The three items I had deferred from the review as too big for a quick pass, done
deliberately with a before-and-after QEMU boot to prove behavior did not move.

- Split main.rs. Everything from the REPL down (input handling, command dispatch,
  the cross-dialect jump handler, the ParaASM REPL, and all the Phase-2 handlers)
  moved into a new repl.rs. main.rs now holds only what belongs at the crate root:
  the bump allocator, the bare-metal entry (rust_start / kmain), the boot banner,
  and the panic handler. It went from a monolith to a small entry-point file. The
  only cross-boundary change is that repl() is now pub and kmain calls it through
  the module; nothing else referenced anything that moved, so nothing else changed.
  I trimmed the imports on both sides to stay warning-clean.

- Hardened the FSPLIT fork cap in the kernel. The old code silently dropped any
  fork nested past the fixed depth, which would desync the matching FFUSE. I raised
  the cap and, more to the point, made overflow observable: it increments a
  fork_overflow counter instead of vanishing. Real programs never nest that deep,
  so tick semantics are unchanged and this only removes the silent-failure mode. I
  did NOT touch the join semantics of the verified Frobenius loop.

- Untracked IG_catalog.json. It is a tuple-provenance dump duplicated from the
  canonical imscribing_grammar copy, and the no_std kernel never reads it at runtime
  (the compiled catalog.rs is the runtime source). git rm --cached plus a .gitignore
  entry keeps it on disk and in history while taking the weight out of the tracked
  tree. Reversible in one command if I want it tracked again.

Verified: the release build is error-clean and warning-clean on the pinned nightly,
and a scripted QEMU boot (help, status, ig, quit) produces output byte-for-byte
identical to the pre-refactor baseline.
